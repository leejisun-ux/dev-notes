API 기초
기초 학습API · HTTP 기초

API란?

두 서비스가 서로 "대화"할 수 있게 해주는 통로. n8n과 Gmail, Gmail과 Claude가 데이터를 주고받는 방법이 바로 API예요. API가 없으면 서비스끼리 소통 불가 → 자동화도 불가능.

API와 HTTP의 관계

이 둘은 자주 같이 나와서 같은 말처럼 느껴지지만, 역할이 달라요.

📋
API — 대화 규칙
"이 주소로 이렇게 요청하면, 이런 데이터를 돌려줄게"라는 약속이에요. 어떤 수단으로 전달할지는 상관 안 해요.
🚚
HTTP — 전달 수단
인터넷에서 데이터를 실어 나르는 배송 시스템이에요. API의 요청/응답을 실제로 전달하는 역할을 해요.
💡
비유: API = 배송 주소 규칙 + 계약서, HTTP = 택배 트럭
대부분의 API가 HTTP로 작동해서 거의 같은 말처럼 쓰이지만, 엄밀히는 HTTP가 API보다 더 하위 개념이에요.
전체 흐름 한 줄
React 화면 → 버튼 클릭 → HTTP로 전달 → API 규칙대로 요청 → 서버 처리 → JSON으로 응답 → 화면에 표시
HTML/CSS/JS·React는 화면 담당  ·  JSON은 데이터 포장 형식  ·  HTTP는 전달 수단  ·  API는 대화 규칙

HTTP 요청의 4가지 구성요소

🎯
Endpoint
정확한 주소. 어떤 서비스의 어떤 기능을 쓸지 명시. 주소가 틀리면 요청 실패.

api.replicate.com/v1/models/.../predictions
📋
Headers
택배 송장처럼 부가 정보를 담아요. 인증(API 키), 데이터 형식 등을 명시.
📬
Method
어떤 방식으로 요청할지. GET(가져오기) / POST(보내기)를 90% 이상 사용.
📝
Body
상세 요청 정보. POST 요청에서 주로 사용. JSON 형식으로 작성.
⚠️ API 키 = 비밀번호. 타인이 내 키로 요청하면 내 비용으로 청구돼요. 절대 공개하면 안 됨!
Method 종류
GET
데이터를 읽어옴. 서버에 변화 없음
예) 날씨 조회, 이메일 목록
POST
새 데이터 생성 또는 작업 실행
예) AI 이미지 생성, 메시지 발송
PUT / DELETE
수정 / 삭제. 사용 빈도 낮음.
현업에선 GET·POST가 90%+
Body 예시 (JSON)

"파스타 주세요"가 기본 요청이라면, Body는 "면 곱빼기, 마늘 추가, 치즈 빼주세요" 같은 상세 옵션.

{
  "input": {
    "prompt": "Make the sheets in the style of the logo",
    "image_input": [이미지 URL 배열],
    "aspect_ratio": "match_input_image",
    "output_format": "jpg"
  }
}
전체 흐름
1단계 — 문서 확인
  • 어떤 Endpoint가 있는지
  • 어떤 Method를 써야 하는지
  • 어떤 Headers가 필요한지
  • Body에 뭘 넣어야 하는지
2단계 — 요청 구성
  • Endpoint URL 설정
  • Method 선택 (GET/POST)
  • Headers에 API 키 입력
  • Body에 JSON 데이터 작성
3단계 — 전송 & 수신
  • 서버가 요청 처리
  • JSON 형식으로 결과 반환
  • 성공 또는 에러 확인

Parameters 채우는 2가지 방법

방법 1 — Claude에게 물어보기

2

방법 2 — curl import 사용 (더 빠름!)
1
Replicate에서 테스트 실행
2
Recent predictions 클릭 → HTTP 복사
3
n8n에서 [import curl 버튼] 클릭 → 붙여넣기 → 자동으로 설정 완성!
✨ 테스트 후 자동으로 설정이 채워지므로 방법 1보다 훨씬 빠름

LoRA란?

Low-Rank Adaptation의 줄임말. AI 이미지 모델 전체를 처음부터 다시 학습시키지 않고, "이 스타일만 추가로 학습"시키는 가벼운 방법이에요.

🎨
비유: 이미 그림을 잘 그리는 화가(기반 모델)에게 "우리 회사 스타일 샘플 30장"을 보여주고, 그 스타일만 따로 학습시키는 것

기반 모델 vs LoRA

구분기반 모델LoRA
크기
수 GB ~ 수십 GB
수 MB ~ 수백 MB
학습 비용
매우 비쌈
상대적으로 저렴
용도
이미지 생성 전반
특정 스타일·캐릭터 학습
예시
Flux, SDXL, Stable Diffusion
브랜드 스타일 LoRA ← 우리 케이스!

1

2
캡셔닝(설명 달기) — 각 이미지에 텍스트 설명을 달아줌. "brand illustration style, flat color, geometric shape" 등
3
학습 실행 — Fal.ai 또는 Replicate에 업로드 → LoRA 학습 시작 (수십 분 ~ 수 시간)
4

5
API 연결 — 학습된 모델의 API 엔드포인트를 웹앱에 붙이기

Fal.ai vs Replicate 비교

⚡

지금 이 파일의 정체

지금 보고 있는 이 학습장은 HTML 파일 하나예요. 서버도 없고, 데이터베이스도 없어요. 브라우저가 파일을 열어서 보여주는 것뿐이에요.

💡
그래서 어디든 올려놓고 URL로 열면 바로 쓸 수 있어요. 가장 쉬운 방법은 GitHub + Vercel 조합이에요.

📦 추천 방법: GitHub + Vercel

✅
둘 다 무료. 한 번 설정하면 파일 수정하고 저장할 때마다 자동으로 반영돼요.
1
GitHub 가입 — github.com 에서 가입 (이미 있으면 생략)
2
새 Repository 만들기 — "dev-notes" 같은 이름으로. Public으로 설정
3
HTML 파일 업로드 — 파일 이름을 반드시 index.html 로 바꾸고 업로드
4
Vercel 가입 — vercel.com → "Continue with GitHub"으로 로그인
5
Import 하기 — "Add New Project" → 방금 만든 GitHub Repository 선택 → Deploy 버튼
6
완료! — https://dev-notes-xxx.vercel.app 같은 URL이 생겨요. 즐겨찾기 해두면 끝

코딩 개념 지도
📚 학습 노트전체 개념 관계도

헷갈리는 코딩 개념들을 6개 레벨로 재분류했어요. 같은 레벨끼리는 형제 관계, 위아래는 부모-자식 관계예요.

Level 1  개발 대상    웹개발 / 앱개발
Level 2  시스템 구조  프론트엔드 / 백엔드 / 서버 / 데이터베이스
Level 3  통신 프로토콜 HTTP / API / Webhook / MCP
Level 4  프로그래밍 언어  HTML / CSS / JS / Python…
Level 4.5 데이터 구조   JSON / Schema / 변수 / Property / Parameter
Level 5  프레임워크 & 라이브러리 React / MUI / Storybook…
Level 6  도구 & 서비스  Cursor / GitHub / Vercel / Replicate / n8n…
L1 개발 대상 — 뭘 만드나?
웹개발
브라우저에서 열리는 것. URL 있으면 다 웹. 이 학습장도 웹이에요.
앱개발
폰에 설치하는 것. iOS/Android 앱. 웹과 목적지만 달라요.
💡
코딩 = 요리하는 행위 자체. 웹개발/앱개발 = 양식당 할지 배달앱 할지 결정하는 것.
L2 시스템 구조 — 스마트 레스토랑 비유
🏠
프론트엔드(홀) + 백엔드(주방) + 서버(건물) + DB(창고) — 다 같은 식당 안에 있지만 역할이 달라요.
프론트엔드
손님이 보는 홀. 화면, 버튼, 색깔 — 눈에 보이는 것 전부. HTML·CSS·JS·React로 만들어요.
백엔드
손님 못 보는 주방. 데이터 처리, API 연결, 인증 담당. 프론트가 요청하면 주방이 처리해요.
서버
식당 건물 자체. 백엔드 코드가 24시간 돌아가는 컴퓨터. Vercel이 이 건물을 대신 관리해줘요.
데이터베이스
식재료 창고. 모든 데이터를 저장하는 곳. Supabase가 대표 서비스예요.
스키마
창고 선반 설계도. "고기 칸", "채소 칸" 라벨처럼 DB에 데이터를 어떤 형태로 저장할지 미리 정한 구조.
L3 통신 개념 — 우주 배달 서비스 비유
🚀
HTTP(우주 항로) → API(우주선) → Endpoint(행성 좌표) → Method(깃발) → Headers/Body(운송장+내용물)
HTTP
우주 항로. 인터넷에서 데이터가 이동하는 기반 규칙. 모든 웹 통신이 이 위에서 일어나요. Hypertext Transfer Protocol.
HTTP를 구성하는 요소들: Endpoint(행성 좌표/정확한 주소) · Method(행동 종류) · GET(가져오기 🚩) · POST(제출하기 🚩) · Headers(운송장/인증 정보) · Body(상자 안 내용물) · Token/Bearer(VIP 출입증)
API
우주선/웨이터. "이 주소로 이렇게 요청하면 이걸 돌려준다"는 약속과 통로. HTTP 항로를 타고 움직여요.
API 문서
웨이터 매뉴얼. API는 실제 통로(문), 문서는 그 문 여는 방법 설명서. 둘은 다른 것이에요!
SDK
밀키트. API(원재료)를 다루기 쉽게 다듬어놓은 것. API는 직접 HTTP 요청을 짜야 하지만 SDK는 한 줄로 끝나요.
Webhook
초인종/자동 알림벨. API는 내가 먼저 요청, Webhook은 서비스가 먼저 알림. 방향이 반대예요.
MCP
AI용 만능 열쇠. AI 비서가 여러 서비스(도구)를 표준화된 방식으로 바로 쓸 수 있게 해주는 인터페이스. API가 개별 공구라면 MCP는 공구함 열쇠.
파라미터
주문서 세부항목. "짜장면 곱빼기, 단무지 많이" 같은 상세 조건값. API 요청에 넣는 옵션들이에요.
L4 프로그래밍 언어 — 요리 언어들
📝
언어들은 부모-자식 관계가 아니에요. 목적이 다른 독립적인 언어들이에요. JSON은 언어가 아니라 데이터 형식이에요.
HTML
뼈대. "여기 제목, 여기 버튼" 구조 담당. 웹의 기초.
CSS
인테리어. 색깔, 크기, 레이아웃. HTML 뼈대에 입히는 옷.
JavaScript (JS)
두뇌. 버튼 클릭, 데이터 처리, API 호출 — 모든 동작 담당.
Python
만능 언어. AI/데이터에 압도적. 문법이 쉬워서 입문자 친화적.
Java
기업용. 대형 서버·안드로이드 앱에 쓰임. 지금 프로젝트엔 불필요.
JSON → L4.5로 이동
언어가 아닌 데이터 포맷. 아래 L4.5 데이터 구조에서 자세히 다뤄요.
마크다운
간단한 메모법. # 제목, **굵게** 같은 기호로 글 꾸미기. 개발 문서에 많이 씀.
.env
금고. API 키 등 비밀값 보관. 코드와 분리해서 GitHub에 절대 올리면 안 됨.
웹 3형제 쌓이는 순서: HTML(뼈대) → CSS(옷) → JS(두뇌) — 이 순서로 레이어가 쌓여요.
L4.5 데이터 구조 — 데이터를 담고 정리하는 방식
📦
언어도 아니고 도구도 아닌, 데이터를 어떻게 담고 구조화하는지에 관한 개념들이에요.
JSON
공용 택배 상자 규격. 언어 ❌ 데이터 포맷 ✅. 어떤 언어로든 읽을 수 있는 데이터 형식. API 응답이 거의 다 JSON이에요.
Schema
창고 선반 설계도. DB에 데이터를 어떤 형태로 저장할지 미리 정한 구조. "이름은 텍스트, 나이는 숫자" 같은 것.
변수
이름표 붙은 상자. userName = "지선" 처럼 값을 담아두는 그릇. 코드 어디서든 꺼내쓸 수 있어요.
Property
물건의 속성. 버튼의 색깔, 크기, 텍스트 같은 특성값. React에서는 props라고 불러요.
Parameter
주문서 세부항목. "짜장면 곱빼기, 단무지 많이" 처럼 API 요청에 넣는 상세 조건값. 함수에 넣는 입력값이기도 해요.
L5 프레임워크 & 라이브러리 — 이케아 시스템
🛋
라이브러리 = 이케아 시스템 전체 / React = 이케아 조립 방식 / MUI = 이케아 주방 수납장 세트.
피그마 UI Kit처럼 — 버튼 픽셀 단위로 안 그리고 Kit 가져다 쓰는 것과 똑같아요!
라이브러리 (전체 집합)
↳ React (특정 브랜드/방식 — JS로 화면 효율적으로 그리는 라이브러리)
↳ MUI (React 전용 구글 디자인 UI 부품 세트)
↳ Framer Motion (React 전용 애니메이션 세트)
↳ Lucide React (React 전용 아이콘 세트)
↳ Storybook (UI 컴포넌트 전시실 — React와 함께 씀)
라이브러리
조립 가구 키트. 누군가 미리 만든 코드 모음. 처음부터 안 짜고 가져다 써요.
프레임워크
아파트 (라이브러리와 차이). 라이브러리는 내가 필요할 때 부르는 도구, 프레임워크는 그 안에 들어가서 규칙대로 살아야 하는 구조.
L6 도구 & 서비스 — 실제로 쓰는 것들
🛠
설치하거나 가입해서 쓰는 실제 제품들이에요. 개념이 아니라 서비스.
IDE — 코드 짜는 주방 조리대
Cursor
AI 내장 코드 에디터. 바이브코딩 핵심. VS Code 기반.
Claude Code
터미널에서 쓰는 Claude. 파일 직접 수정 가능. 에이전트 방식.
Codex
OpenAI 코딩 AI. Cursor 안에서 모델로 선택 가능.
Antigravity
국내 바이브코딩 도구. Cursor와 비슷한 포지션.
버전관리 — 게임 세이브 포인트
Git
시스템(개념). 코드 변경 이력 추적. "타임머신 소프트웨어".
GitHub
서비스(제품). Git 기록을 올리는 온라인 창고. Git ≠ GitHub.
인프라 & AI 엔진
Vercel
서버 + 배포 플랫폼. 코드를 올리면 자동으로 서버에 올려서 공개까지 해줘요. 서버 직접 안 다뤄도 됨.
Supabase
데이터베이스 서비스. 바이브코딩 프로젝트의 단골 창고.
Replicate
AI 모델 빌려쓰기. LoRA 모델 올려두고 API 호출. 모델 종류 다양.
n8n
노코드 자동화. "A 발생하면 B 해줘" GUI로 만드는 것. 레고 블록 조립.

🤯 가장 헷갈리는 개념 TOP 10

01
API vs API 문서 — API는 실제 문, 문서는 그 문 여는 방법 설명서. 둘은 다른 것.
02
Git vs GitHub — Git은 내 컴퓨터의 타임머신 소프트웨어, GitHub은 그 기록을 올리는 온라인 보관소.
03
라이브러리 vs 프레임워크 — 라이브러리는 내가 필요할 때 부르는 도구, 프레임워크는 그 안에서 규칙대로 살아야 하는 아파트.
04
JSON vs 언어 — JSON은 언어가 아니라 공용 데이터 포맷(택배 상자 규격). 어떤 언어로든 읽을 수 있어요.
05
Vercel vs 서버 — 서버는 개념적인 컴퓨터, Vercel은 그 컴퓨터를 클릭 몇 번으로 빌려주는 임대 서비스.
06
GET vs POST — GET은 정보 가져오기(주소창에 노출), POST는 정보 제출하기(숨겨서 보냄).
07
Token vs API Key — API Key는 현관 비밀번호(영구), Token은 일정 시간만 유효한 임시 출입증.
08
SDK vs API — API는 원재료, SDK는 그걸 다루기 쉽게 다듬어놓은 밀키트.
09
터미널 vs IDE — IDE는 화방(그림 그리는 공간), 터미널은 화방 사장님께 문자로 명령 내리는 인터컴.
10
n8n vs 코딩 — n8n은 레고 블록 조립(GUI), 코딩은 레고 블록 자체를 깎아 만드는 작업.
